## Design Context

### Users

There are **two audiences** only—no separate “marketing personas” in product scope:

- **Clients** — People applying for a visa (e.g. UAE tourist visa): landing, auth, apply, and portal under `app/(client)/**`. Job to be done: **understand options, submit documents, pay, and track status** with minimal friction.
- **Admins** — Internal **ops staff** who run the site: queues, catalog/pricing, verification, and related tools under `app/admin/**` (and any admin-scoped portal routes described in the PRD).

### Brand Personality

- **Client (Visatop):** **Professional, easy to use, and modern**—implemented with the Visatop marketing system (ink, blue structure, yellow CTAs, Inter + Noto Serif) so the app feels **continuous with the company site**, not like a generic template.
- **Admin:** **Structured, high-contrast, operational**—Red Hat–inspired clarity (per [`PRODUCT_REQUIREMENTS.md`](PRODUCT_REQUIREMENTS.md)), sharp hierarchy; supports **light and dark** themes.

### Aesthetic Direction

- **Two skins, shared primitives:** Shared shadcn/ui + Tailwind; **client** uses scoped `.theme-client` (light-only), **Inter** + **Noto Serif** via `app/(client)/layout.tsx`; **admin** keeps **Red Hat** fonts from the root layout and `:root` / `.dark` tokens.
- **Client application UX benchmark:** Stepwise, direct **apply online** flows in the spirit of high-trust consumer visa portals—for pattern reference, see [Dubai Visa Online](https://dubaivisa.com/) (nationality → product choice → clear requirements → pay/submit). Match **structure and clarity**, not a pixel-perfect clone; stay on Visatop tokens and components.
- **Client tokens (summary):** Ink `#012031`, primary blue `#224D64`, surface `#F2F9FC`, CTA yellow `#FCCD64`, links/accent gold `#CE8E00`, fast linear hovers, **5px** control radius baseline, **10–12px** on larger surfaces, selective **cut-corner** panels for hero/highlight—not everywhere. Full mapping: [`website-brand-guidlines-summary.md`](website-brand-guidlines-summary.md) and the [client brand theme spec](docs/superpowers/specs/2026-04-20-client-brand-theme-design.md).
- **Non-goals for client:** No client dark mode in scope; do not bleed client variables into admin routes.

### Design principles

1. **Split the source of truth by surface** — **Clients** follow Visatop website guidelines + client theme spec; **structure and IA** still align with [`PRODUCT_REQUIREMENTS.md`](PRODUCT_REQUIREMENTS.md) where it describes screens and flows. **Admins** follow this PRD for layout, density, palette, and typography (Red Hat stack).
2. **Scope brand by route tree** — Client look-and-feel applies under the client layout only; admin remains visually isolated and theme-toggle-capable.
3. **Semantic tokens first** — Map brand into shadcn CSS variables (`--primary`, `--muted`, etc.); avoid scattering hex in components. Use thin **client wrappers** only where tokens cannot encode a pattern (nav underline, cut corners, prescribed shadows).
4. **Clarity over novelty** — Visa flows need predictable hierarchy, readable **18px** body on client, obvious primary actions, and calm error/validation states (`--error` vs destructive red where the spec distinguishes them).
5. **Desktop-first density where operators work** — Admin patterns stay data-dense; client flows stay scannable and forgiving on smaller viewports where applicants apply.
6. **Motion is subtle and fast** — Prefer short, purposeful transitions; respect `prefers-reduced-motion` when adding motion.

### Accessibility

Ship to a **strong default**: semantic HTML, visible focus, keyboard operability for interactive controls, labels tied to inputs, and color contrast that targets **WCAG 2.2 AA** for normal text and UI components unless a deliberate exception is documented. Honor **`prefers-reduced-motion`** for non-essential animation. Prefer accessible names and live regions for async status (upload, payment, errors).
